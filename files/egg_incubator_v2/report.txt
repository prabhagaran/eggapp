C-1  CLOUD_TOKEN stored in plaintext in config.h
     File: config.h, line ~232
     #define CLOUD_TOKEN  "a1b2c3d4e5f60718293a4b5c6d7e8f90"
     This secret is committed to source code.  Anyone with repo access can
     replay requests to the Google Apps Script endpoint.
     FIX: Move to a gitignored "secrets.h" (and add "secrets.h" to .gitignore).
     Alternatively, replace with an HMAC-SHA256 signature over the timestamp
     so the token rotates per request.

C-2  GOOGLE_SCRIPT_URL hardcoded in config.h
     File: config.h, line ~229
     The deployment URL is the stable endpoint identifier.  Exposing it in
     source enables anyone to discover it and flood the sheet.  This is less
     critical than C-1 but should be moved to secrets.h alongside CLOUD_TOKEN.

C-3  TLS verification disabled — CLOUD_ROOT_CA is empty
     File: config.h line ~220, task_cloud.cpp throughout
     CLOUD_ROOT_CA "" causes all three WiFiClientSecure instances to call
     setInsecure(), making HTTPS connections vulnerable to MITM attacks.
     All telemetry data and the CLOUD_TOKEN travel over an unverified channel.
     FIX: Populate CLOUD_ROOT_CA with Google's ISRG Root X1 PEM certificate
     (or *.googleapis.com / Google Trust Services chain as appropriate for
     script.google.com / redirect targets).

H-1  RELAY_FAN / LEDC pin conflict — gRelayState.fanOn always false
     Files: globals.cpp (setRelay, allRelaysOff), task_incubator.cpp (setFanSpeed)
     RELAY_FAN (GPIO 13) is configured as an LEDC PWM output by setFanSpeed().
     Other code paths (allRelaysOff, setRelay) call digitalWrite(RELAY_FAN, …),
     which on ESP32 Arduino framework detaches LEDC from the pin and reverts it
     to standard GPIO, silently breaking PWM output.
     Additionally, setFanSpeed() never calls setRelay(), so gRelayState.fanOn
     stays false permanently.  Cloud telemetry always reports fan=0.
     FIX (option A — recommended): Remove RELAY_FAN from allRelaysOff() and
     setRelay() entirely.  Use setFanSpeed(0) as the only off-path.  Update
     gRelayState.fanOn manually in setFanSpeed() under controlMutex.
     FIX (option B): Dedicate a separate GPIO for the relay enable/disable, keep
     LEDC on the PWM pin, and control the relay gate with that GPIO.

H-2  Turner records lastTurnEpoch = nowEpoch (read before turn), not after turn
     File: task_incubator.cpp, task_turner()
     The epoch read before waiting durationSec is used as lastTurnEpoch.
     This means the real idle time after the turn is
       turnerIntervalMin - turnerDurationSec  (e.g. 3 h 59 m 30 s instead of 4 h).
     For short durations (30 s) and long intervals (4 h) the error is ~0.2 %.
     For tighter settings (e.g. 10-min interval, 2-min turn) the error is 20 %.
     FIX: Re-read nowEpoch after  vTaskDelay(durationSec * 1000) and use that
     as the new lastTurnEpoch.

H-3  Climate cycleStartEpoch and ramp state not persisted to NVS
     Files: climate_logic.cpp cyclicInHeatPhase(), getRampTargetTemp()
     Both functions update gSettings.cycleStartEpoch / rampStepIdx /
     rampStepStartEpoch in RAM only.  A power cycle resets cyclic and ramp
     positions to zero, restarting the full cycle from scratch.
     FIX: After writing gSettings.cycleStartEpoch in cyclicInHeatPhase()
     bootstrap, write a single NVS key ("cycleStart") via Preferences, mirroring
     the pattern used for lastTurn in task_incubator.cpp.  Do the same for
     rampStepIdx and rampStepStartEpoch in getRampTargetTemp().

H-4  Lockdown humidity change not persisted to NVS
     File: task_incubator.cpp, task_milestone()
     On lockdown day, gSettings.humSetpoint is raised to lockdownHumidity in
     RAM.  Power cycle reverts humSetpoint to the pre-lockdown value, and the
     humidifier under-targets until the user re-notices.
     FIX: Write prefs.putFloat("setHum", newValue) immediately after the
     in-memory update, or call saveSettings() inside task_milestone() (noting
     that saveSettings() holds settingsMutex for a blocking NVS write — consider
     a small dedicated helper that writes only the hum key).

H-5  task_pump() uses possibly stale / invalid humidity without validity check
     File: task_incubator.cpp, task_pump()
     currentHum is read from gSensorData.humidity_dht but gSensorData.hum_valid
     is never checked.  If the DHT sensor has consecutive failures, the DHT task
     holds the last valid value in gSensorData.humidity_dht but sets
     hum_valid = false.  The pump task still acts on that stale value and could
     trigger an erroneous pump run.
     FIX: Read hum_valid in the same sensorMutex critical section and skip the
     pump logic when it is false.

M-1  Orphaned defines DEFAULT_FAN_INTERVAL_MIN and DEFAULT_FAN_DURATION_SEC
     File: config.h lines ~119–120
     These were meaningful before the fan was refactored to PWM-only control.
     They are now unreferenced anywhere.  They mislead future readers.
     FIX: Remove both defines.  The comment block already says "FAN DEFAULTS";
     keeping only DEFAULT_FAN_SPEED_PERCENT is sufficient.

M-2  TelemetryMsg_t declared after its extern in globals.h
     File: globals.h lines ~228–242
     The extern QueueHandle_t telemetryQueue; declaration precedes the
     TelemetryMsg_t typedef.  While this compiles (the queue handle does not
     depend on the struct), it breaks the logical "type before usage" convention
     and could confuse static analysers.
     FIX: Move the TelemetryMsg_t typedef above the extern declarations block.

M-3  Candling day is hardcoded to day 7 for all egg types
     File: incubator_logic.cpp, checkMilestone()
     "if (day == 7)" is applied regardless of EggType.  Quail (17-day) and
     duck (28-day) have different candling schedules.
     FIX: Add per-type constants to config.h (e.g. CHICKEN_CANDLE_DAY 7,
     DUCK_CANDLE_DAY 10, QUAIL_CANDLE_DAY 7) and look them up from gSettings.eggType,
     or derive it as (totalDays / 3).

M-4  SENSOR_INVALID_HIGH = 85.0f clashes with DS18B20 POR sentinel + climate range
     File: config.h line ~64
     The DS18B20 power-on reset value (85.0 °C) is explicitly caught in
     task_sensors.cpp (if t == 85.0f).  However, SENSOR_INVALID_HIGH = 85.0f
     means temperatures strictly above 85 °C (e.g. 86 °C) are also marked invalid,
     which could silently drop real readings in a high-temperature climate chamber.
     The climate fault threshold (MAX_TEMP_CLIMATE = 80.0 °C) would already trigger
     before 86 °C can appear, so in practice this is benign — but the semantics
     are confusing.
     FIX: Rename to SENSOR_INVALID_HIGH 100.0f (or sensor spec limit) and rely on
     the explicit == 85.0f check for DS18B20 POR separately.

M-5  TURNER_MIN_INTERVAL_MIN comment is wrong
     File: config.h line ~106
     #define TURNER_MIN_INTERVAL_MIN  1  // 60 min minimum allowed
     The comment says "60 min" but the value is 1.  Either the value or the
     comment is wrong.  A 1-minute turner interval would stress the motor and
     is physiologically pointless.  The comment likely reflects an intended
     business rule.
     FIX: Confirm intent and update.  Suggested: #define TURNER_MIN_INTERVAL_MIN  60

M-6  switchProfile() may deadlock if called while a task holds settingsMutex
     File: egg_incubator_v2.ino, switchProfile()
     switchProfile() first calls allRelaysOff() (acquires controlMutex), then
     takes settingsMutex to update activeProfile, and finally calls
     vTaskSuspend(hTaskTempControl) / vTaskSuspend(hTaskTurner) etc.
     If any of those tasks is currently blocked waiting for controlMutex or
     settingsMutex, suspending the task while it holds a mutex leaves the mutex
     permanently locked.
     FIX: Ensure allRelaysOff() and the settingsMutex section complete before
     any vTaskSuspend calls.  The current code order does this correctly only if
     the suspended tasks are not mid-critical-section.  Add a brief vTaskDelay
     after setting the profile flag to let tasks reach their next wait point
     before suspension (a 100 ms delay is already present — confirm it is
     sufficient for all task periods).

M-7  Cloud retry loop iterates snapshot count but requeues inside the loop
     File: task_cloud.cpp lines ~45–82
     size_t queued = uxQueueMessagesWaiting(telemetryQueue) is captured once.
     Messages that are not yet due are placed back onto the same queue.
     Because FreeRTOS queues are FIFO, a message may be dequeued and re-enqueued
     N times in one pass if the queue fills up with back-pressure items.
     In practice with TELEMETRY_QUEUE_SIZE = 16 and the backoff guard this is
     unlikely to cause a visible problem, but the loop should process *at most*
     the number of messages that are currently due, not all queued messages.
     FIX: Only attempt a send if (int32_t)(msg.nextAttemptMs - millis()) <= 0;
     otherwise stop processing for this cycle (the items not due will stay in
     the queue for the next iteration).

M-8  wifiUserEnabled / wifiPortalActive are volatile but unguarded across tasks
     File: globals.h / globals.cpp
     These flags are written from task_wifi_manager (Core 0) and read from
     task_cloud (Core 0) and task_ui (Core 1).  On ESP32 with dual Xtensa cores,
     volatile alone does not guarantee atomicity for concurrent reads and writes
     from different cores — the store/load may be re-ordered by the hardware
     without a memory barrier.  A portENTER_CRITICAL or atomic_bool wrapper
     would be safer.

LOW  (code quality, maintainability, or minor style)
────────────────────────────────────────────────────────────────────────────────

L-1  task_ui.cpp is 1,493 lines — split recommended
     The UI task handles 30+ states in a single file.  Each logical screen group
     (Home, Environment, Climate, Settings, Time/Date) could be a separate
     module with a common handle_xxx(UiEvent) entry point.

L-2  TelemetryMsg_t.url[768] — verify URL does not exceed buffer
     The telemetry URL includes ~155-char base URL + ~250 chars of parameters
     after URL-encoding.  The current worst case (Climate profile with all fields)
     fits well under 768 bytes, but no runtime check or snprintf limit exists.
     FIX: Use snprintf with sizeof(t.url) when building the URL, and log a
     warning if truncation occurs (check return vs buffer size).

L-3  pushError silently drops messages when errorQueue is full
     File: globals.cpp, pushError()
     xQueueSend with timeout 0 drops silently.  In high-error scenarios, error
     messages may be lost without any indication.
     FIX: Add a volatile overflow counter (droppedErrors++) and periodically
     transmit it as a meta-error or display on the OLED fault screen.

L-4  Serial output inside sensorMutex / settingsMutex critical sections
     Multiple files (e.g. task_control.cpp) print Serial.printf for heater ON/OFF
     while holding or immediately after releasing mutexes.  Serial output takes
     variable time and on ESP32 can block for milliseconds under heavy load.
     This is fine for development but should be removed or gated by a
     SERIAL_DEBUG compile-time flag for production firmware.

L-5  DS18B20 uses getTempCByIndex(0) — assumes exactly one sensor on the bus
     File: task_sensors.cpp
     If a second DS18B20 is accidentally connected (or a wire shorts), index 0
     may address the wrong sensor silently.  Using a specific device address
     (saved in NVS during provisioning) would be more robust.

L-6  RTC is checked only for isrunning() at boot; not verified during operation
     File: egg_incubator_v2.ino, setup()
     If DS1307 loses power (no backup battery) or loses I2C during operation,
     gRtcTime.epoch becomes stale and all epoch-dependent logic (turner, milestone,
     incubation day) silently uses bad values.
     FIX: In task_rtc(), validate that the returned epoch is > a minimum sane
     value (e.g. > 1700000000, ~ Nov 2023) and set a flag / push an error when
     it is not.

L-7  No watchdog on critical tasks
     No TWDT (task watchdog timer) enrollment is visible.  If task_temperature_control
     or task_climate_control stalls (e.g. waiting forever on a mutex due to M-6
     above), heaters could remain on indefinitely until the over-temp hardware
     fuse trips.
     FIX: Enrol temperature control tasks in TWDT with a 5-second timeout, or
     ensure all mutex waits have bounded timeouts (they already use 50/100 ms
     timeouts — verify the code paths covering all exits).

L-8  Factory reset does not turn relays off before reboot
     File: egg_incubator_v2.ino, factoryReset()
     prefs.clear() → 500 ms delay → ESP.restart() with no allRelaysOff() call.
     If heater was on, it stays on for 500 ms plus the ESP32 boot time (~500 ms).
     FIX: Call allRelaysOff() as the first line of factoryReset().

L-9  milestoneMutex created inside task_milestone(), not in setup()
     File: task_incubator.cpp
     milestoneMutex = xSemaphoreCreateMutex() is called at the start of
     task_milestone().  Any code that calls checkMilestone() or reads
     gMilestoneLabel before that task runs for the first time will receive
     a nullptr mutex.  The home screen OLED read of gMilestoneLabel in task_ui
     should either be guarded by a null-check or the mutex should be created
     in setup() alongside the other primitives.

L-10 HTTP response body read on every request even when not used
     File: task_cloud.cpp
     String body = http.getString() is called after every GET regardless of
     whether the content is actually used (most paths only log it on success
     for telemetry, error sends ignore data).  On an ESP32 with constrained
     heap, reading a potentially large HTTP body into a String on every request
     wastes RAM. Use http.getString() only when the body is actually inspected.
