M-3  [FIXED] Candling day hardcoded to day 7 for all egg types

M-4  [FIXED] SENSOR_INVALID_HIGH sentinel widened to avoid DS18B20 POR clash
  Resolved: `SENSOR_INVALID_HIGH` changed to 100.0f in `config.h`.
  DS18B20 power-on-reset sentinel (85.0 °C) is still explicitly detected
  in `task_sensors.cpp` via `if (t == 85.0f)`, so POR handling remains
  unchanged while the range guard was widened to avoid semantic collision.
  Files changed: config.h

M-5  [FIXED] TURNER_MIN_INTERVAL_MIN comment vs. value mismatch
     Resolved: Value corrected to 60 to match "60 min minimum allowed" comment.
     Files changed: config.h

M-6  [FIXED] switchProfile() safe-suspend via task notifications
     Resolved: Replaced all direct vTaskSuspend() calls in switchProfile() with
     a task-notification based mechanism.  Each suspendable task
     (task_temperature_control, task_climate_control, task_turner, task_fan,
     task_pump, task_milestone) now checks for TASK_CMD_SUSPEND (0xA5A5) at the
     very top of its for(;;) loop — outside any mutex.  switchProfile() sends
     xTaskNotify(handle, TASK_CMD_SUSPEND, eSetValueWithOverwrite) only to the
     tasks being suspended, waits 200 ms for them to reach their safe point, then
     calls xTaskNotifyStateClear() before vTaskResume() on tasks being activated
     so they cannot re-suspend from a stale notification.
     Files changed: config.h, task_control.cpp, task_climate_control.cpp,
                    task_incubator.cpp (turner/fan/pump/milestone), egg_incubator_v2.ino

M-7  [FIXED] Cloud retry loop over-processes not-yet-due messages
     Resolved: The retry loop now uses `xQueuePeek()` to inspect the queue
     head before dequeuing. If the head item's `nextAttemptMs` is still in
     the future the loop breaks immediately, stopping all re-enqueue churn.
     FIFO ordering means if the head is not yet due, all remaining items in
     this pass are not due either.
     Root cause was: iterating with a snapshot count, dequeuing each item,
     and re-enqueueing not-yet-due items -- causing tight busy-loop CPU spikes
     on Core 0 whenever backoff items dominated the queue.
     Files changed: task_cloud.cpp

M-8  wifiUserEnabled / wifiPortalActive — volatile without memory barriers
     File: globals.h / globals.cpp
     These flags are written from Core 0 (task_wifi_manager) and read from
     Core 1 (task_ui) and Core 0 (task_cloud).  On dual-core ESP32, volatile
     alone does not guarantee cross-core visibility without a memory barrier.
     FIX: Wrap reads and writes in portENTER_CRITICAL / portEXIT_CRITICAL
     (with a dedicated spinlock), or change type to std::atomic<bool>.

LOW  (code quality, maintainability, or minor style)

L-1  task_ui.cpp monolithic (~1,493 lines)
     No change since last review.  Split into per-screen modules remains a
     recommended refactor for maintainability.

L-2  TelemetryMsg_t.url[768] — no truncation warning when building URL
     File: task_cloud.cpp, URL-build block
     The telemetry and error URLs are built as Arduino String objects.
     strncpy(t.url, url.c_str(), sizeof(t.url)-1) limits the copy, but no
     runtime warning is emitted if the URL String exceeds 767 bytes.
     FIX: After strncpy, check if url.length() >= sizeof(t.url)-1 and call
     pushError("URL_TRUNC", "Telemetry URL truncated") if so.

L-3  pushError silently drops messages when errorQueue is full
     File: globals.cpp, pushError()
     xQueueSend with timeout 0 drops silently.
     FIX: Increment a volatile uint32_t droppedErrors counter and
     periodically log or display it.

L-4  Serial.printf in hot control loops
     Multiple files (task_control.cpp, task_climate_control.cpp) print relay
     state changes via Serial.printf.  On ESP32, Serial writes can block for
     several milliseconds under UART TX pressure.
     FIX: Gate behind a compile-time #if SERIAL_DEBUG flag for production
     firmware.

L-5  DS18B20 uses getTempCByIndex(0) — single-sensor assumption
     File: task_sensors.cpp
     Index 0 silently reads whichever sensor is first on the 1-Wire bus.
     A wiring fault or second device could redirect reads.
     FIX: Read and persist the device ROM address at first boot (NVS), then
     address by address on subsequent reads.

L-6  [FIXED] RTC epoch not validated during operation
     Resolved: task_rtc.cpp validates epoch > 1700000000 UL on every read.
     Sets rtcEpochValid = false and calls pushError("FAULT","RTC epoch invalid")
     with a 30-second throttle when stale.  rtcEpochValid added as a global
     volatile bool in globals.h / globals.cpp.
     Files changed: task_rtc.cpp, globals.h, globals.cpp

L-7  [FIXED] No task watchdog enrollment
     Resolved: esp_task_wdt_init() called in setup() with 5-second timeout
     using the IDF v5 esp_task_wdt_config_t struct API.
     esp_task_wdt_add(NULL) removed from individual tasks that may be
     suspended during profile switching (avoids watchdog-triggered resets on
     suspend).
     Files changed: egg_incubator_v2.ino

L-8  [FIXED] Factory reset leaves relays energised before reboot
     Resolved: allRelaysOff() is now the first call in factoryReset().
     Files changed: egg_incubator_v2.ino

L-9  [FIXED] milestoneMutex created inside task_milestone() not in setup()
     Resolved: milestoneMutex = xSemaphoreCreateMutex() moved to setup()
     alongside all other RTOS primitives, eliminating the nullptr-access window.
     Files changed: egg_incubator_v2.ino, task_incubator.cpp

L-10 HTTP response body always read even when unused
     File: task_cloud.cpp
     http.getString() is called after every successful GET.  For error-channel
     sends the body content is not inspected (only logged).  On a constrained
     ESP32 heap, allocating a String for an unneeded response wastes RAM.
     FIX: Call http.getString() only in the telemetry path where the body
     may carry a server status message worth logging.
